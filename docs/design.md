# AI 视觉语音助手设计文档

## 1. 项目目标

本项目实现一款 Web 端 AI 视觉对话助手。用户授权摄像头和麦克风后，应用可以持续预览摄像头画面、识别用户语音、让 AI 参考最近视觉内容生成中文回复，并通过浏览器语音播报回答。

当前实现采用 pnpm monorepo、React、NestJS、MinIO 和通义千问 DashScope compatible API。视觉能力使用 `chat/completions` 多模态接口；语音交互使用浏览器 `SpeechRecognition` 和 `SpeechSynthesis`。视频理解采用“本地 2 FPS 采样 + 帧去重 + 短窗口动作序列分析”的近似视频流方案，不上传连续视频流。

## 3. 当前能力与架构

- `apps/web`：React + Vite 前端工作台，负责摄像头预览、本地 2 FPS 动作采样、帧差去重、语音识别、语音播报和会话 UI。
- `apps/server`：NestJS API 服务，负责 DashScope 调用、MinIO 存储、会话结束指标和备用 Omni WebSocket 入口。
- `packages/shared`：前后端共享类型，包括视觉摘要、会话指标和请求/响应契约。

核心流程：

1. 用户点击连接，前端申请摄像头和麦克风权限。
2. 前端启动摄像头预览和浏览器语音识别。
3. 自动观察模式默认开启，前端每 500ms 本地采样一帧。
4. 前端计算轻量帧指纹，低变化帧只在本地跳过，不进入动作缓冲。
5. 前端保留最近约 6 秒去重后的关键帧，并定期调用 `/api/vision/analyze-sequence` 生成动作时间线。
6. 当前画面分析仍通过 `/api/vision/analyze` 生成单帧视觉摘要，并尝试保存关键帧到 MinIO。
7. 用户语音提问时，前端携带最新视觉摘要；如果问题涉及“刚才、动作、手势、发生了什么”，同时携带最新动作时间线。
8. 后端生成自然语言回复，前端展示并播报。

## 4. 成本控制策略

| 成本控制技巧 | 是否想到 | 实际采用 |
| --- | --- | --- |
| 不上传连续视频，只上传关键帧 | 是 | 已采用。当前没有把视频流直接传给云端。 |
| 前端压缩图片 | 是 | 已采用。关键帧压缩到 768px 宽，JPEG 质量 0.72。 |
| 默认低细节视觉分析 | 是 | 已采用。自动和手动分析默认 `detail: low`。 |
| 自动抽帧帧差忽略 | 是 | 已采用。当前帧与上一帧差异低于阈值时跳过上传。 |
| 本地 2 FPS 动作采样 | 是 | 已采用。2 FPS 只发生在浏览器本地，云端不按 2 FPS 调用。 |
| 多帧动作批量分析 | 是 | 已采用。云端只接收去重后的短窗口关键帧序列。 |
| 自动观察频率限制 | 是 | 已采用。自动视觉请求每分钟最多 6 次。 |
| 稳定画面自动降频 | 是 | 已采用。连续稳定画面从 10 秒观察降到约 25 秒观察。 |
| 页面不可见/AI 播报/后端离线时暂停抽帧 | 是 | 已采用，避免后台浪费请求。 |
| 只保存被分析的关键帧 | 是 | 已采用。MinIO 不保存连续视频流。 |
| 接入原生实时音视频模型 | 是 | 未采用。成本和接入复杂度较高，作为后续增强。 |
| 服务端用户级限流和预算告警 | 是 | 未采用。当前 MVP 主要在前端节流，后续可补服务端强约束。 |

## 5. 关键实现说明

- 自动观察开关默认开启，用户可以在控制栏关闭；该模式同时维护当前画面摘要和最近动作时间线。
- `reason: "interval"` 表示自动观察抽帧；`reason: "manual"` 表示用户手动分析；`reason: "visual-question"` 表示用户提问触发的高优先级抽帧。
- 自动抽帧只更新右侧视觉摘要和对话上下文，不默认写入聊天流，避免消息刷屏。
- 动作序列分析使用最近约 6 秒去重关键帧，默认最多上传 8 张压缩 JPEG。
- 提问时如果视觉摘要超过 30 秒，或问题包含“看、画面、摄像头、这是什么、帮我看”等视觉意图，会先尝试刷新当前帧。
- 提问包含“刚才、动作、手势、做了什么、发生了什么”等动作意图时，会优先使用最近动作时间线。
- 系统提示词要求 AI 基于“最近关键帧摘要”和“最近几秒动作时间线”回答；信息不足、漏帧或摘要可能过期时必须说明不确定。

## 6. 环境变量

关键配置：

```env
OPENAI_API_KEY="DashScope API Key"
OPENAI_BASE_URL="https://dashscope.aliyuncs.com/compatible-mode/v1"
OPENAI_VISION_MODEL="qwen3.5-omni-plus"
OPENAI_REALTIME_MODEL="qwen3.5-omni-plus"
MINIO_ENDPOINT="10.62.110.137"
MINIO_PORT=9000
MINIO_USE_SSL=false
MINIO_ACCESS_KEY="admin"
MINIO_SECRET_KEY="12345678"
MINIO_BUCKET="ai-vision-assets"
```

`OPENAI_REALTIME_MODEL` 当前保留为配置项。实际语音闭环仍使用浏览器语音识别/TTS 和后端 `ConversationModule`。
