# AI 视觉对话助手设计文档

## 1. 项目目标

题目要求开发一款与 AI 对话的应用：打开摄像头与麦克风后，让 AI 能看到摄像头中的视频内容、听到用户说的话，并给予恰当回应。同时需要综合考虑视觉理解准确性、语音交互自然度与流畅性，以及端云协同的成本控制策略。

本项目最终实现的是一个 Web 端 AI 视觉对话工作台。用户可以连接摄像头和麦克风，通过语音或文字向 AI 提问。AI 会结合当前画面、最近几秒动作变化、会话历史和场景角色进行中文回答，并在前端流式显示回复、使用浏览器 TTS 播报。

当前方案不是上传原始连续视频流，而是采用“本地关键帧采样 + 帧差去重 + WebSocket 关键帧传输 + 后端滑动视觉记忆”的近实时视觉方案。这样既能让 AI 持续理解画面变化，又能控制云端多模态调用频率和图片上传成本。

## 3. 交互与产品体验

页面采用三栏工作台结构：

- 左侧是会话区：当前会话状态、历史入口、消息列表和文字输入。
- 中间是主舞台：语音状态、场景模式、摄像头画面和控制栏。
- 右侧是场景助手：当前角色说明、推荐问题、当前观察、最近变化、演示建议和折叠调试指标。

核心交互流程：

1. 用户点击“连接设备”，浏览器申请摄像头和麦克风权限。
2. 前端显示摄像头预览，启动语音识别和自动视觉观察。
3. 自动观察默认开启，前端本地采样、压缩、去重关键帧。
4. 有效关键帧通过 `WS /api/omni/realtime` 发送给后端。
5. 后端按 session 维护最新画面观察和最近动作变化。
6. 用户通过语音或文字提问，问题携带当前场景模式。
7. 后端结合场景角色、视觉记忆、动作变化和历史对话生成回复。
8. 前端流式显示回复，完整生成后使用 TTS 播报。
9. 用户可以停止回复、重新说一遍、调整语速或继续输入下一问。
10. 用户点击“结束会话”后，后端归档历史消息、指标和最后一帧引用。


## 4. 场景角色设计

场景模式会同时影响右侧展示重点和后端对话 prompt。每个内置模式对应一个明确 AI 角色：

| 模式 | AI 角色 | 回答倾向 | 右侧展示重点 |
| --- | --- | --- | --- |
| 通用视觉对话 | 通用视觉对话伙伴 | 先直接回答，再补充不确定点。 | 当前观察、最近变化、推荐追问。 |
| 动作理解助手 | 动作观察员 | 优先用“先……然后……”描述动作变化，不把静态外观误判成动作。 | 最近动作、动作步骤、动作类推荐问题。 |
| 演讲/面试练习助手 | 演讲和面试练习教练 | 给一个表现亮点和一个可改进建议。 | 姿态、视线、手势反馈和改进建议。 |
| 生活提醒助手 | 生活提醒助手 | 先说观察，再给轻量提醒，不夸大风险。 | 物品变化、注意事项和轻量提醒。 |
| 自定义模式 | 用户填写的角色卡 | 按用户定义的角色、任务、风格和边界回答。 | 自定义关注点、推荐问题和演示建议。 |

后端 prompt 明确要求 AI 不在用户可见回复中暴露“视觉摘要”“动作时间线”“基于某时间”等工程词，让回答更自然。

自定义模式第一版保存在浏览器 `localStorage`，用户最多保存 5 个角色卡。角色卡包含名称、适用场景、角色身份、核心任务、回答风格、关注重点、推荐问题、演示建议和边界要求。

## 5. 视觉理解方案

项目没有直接上传原始连续视频。原因是原始视频流成本高、实现复杂、稳定性受模型实时协议和账号权限影响较大。当前采用分层方案：

1. 前端摄像头预览保持实时。
2. 本地以低频率采样关键帧。
3. 计算帧指纹和帧差，静止画面不进入上传队列。
4. 对关键帧降尺寸、压缩 JPEG。
5. WebSocket 推送 `video_frame` 到后端。
6. 后端维护 session 级滑动视觉记忆。
7. 后端节流调用多模态模型生成当前画面观察和动作变化。
8. 用户提问时优先读取后端最新视觉记忆，必要时回退到前端缓存或 HTTP 分析。

这种方案在体验上接近“持续观察”，但成本和稳定性比原始视频流更可控。

## 6. 语音体验设计

语音交互基于浏览器 SpeechRecognition 和 `speechSynthesis`：

- 正在听：用户可以直接说话，也可以打字。
- 已听到：识别到最终语音文本后短暂显示。
- 正在思考：后端正在结合画面和问题生成回复。
- 正在回复：文本流式出现，完整生成后开始 TTS 播报。

语音体验增强点：

- “停止回复”可中断 TTS 和流式生成。
- “重新说一遍”只重播本地缓存回复，不增加模型调用。
- “语速”滑杆支持 `0.5x` 到 `3.0x`，适配不同演示环境和用户习惯。
- TTS 目前在完整回复生成后播报，避免分段播报造成断句不自然。

## 7. 历史归档设计

会话过程中，后端按 `sessionId` 维护内存历史，用于连续追问。点击“结束会话”时进行持久化归档：

- `Session`：保存会话 id、状态、开始时间、结束时间。
- `Message`：保存用户和 AI 消息内容、角色和时间。
- `SessionMetric`：保存语音时长、视觉请求数、低/高细节请求数、上传图片字节数。
- `VisionSnapshot`：保存最后一帧图片的 MinIO 对象引用和哈希信息。

图片不直接进入数据库。前端结束会话前抓取最后一帧，后端上传到 MinIO，数据库只保存对象引用。MinIO 不可用时仍允许结束会话和保存文字历史。

历史查看第一版是只读全局弹层：用户可以查看结束时间、消息数量、视觉请求数、最后一帧预览和消息列表。查看历史不会中断当前摄像头、麦克风或视频观察。

## 8. 端云协同与成本控制策略

| 成本控制技巧 | 是否想到 | 实际采用 |
| --- | --- | --- |
| 不上传原始连续视频，只上传关键帧。 | 是 | 已采用。 |
| 浏览器本地低频采样。 | 是 | 已采用，本地采样不等于云端同频调用。 |
| 本地帧差去重。 | 是 | 已采用，低变化帧会跳过。 |
| 上传前压缩图片。 | 是 | 已采用，关键帧会降尺寸并压缩为 JPEG。 |
| 默认低细节视觉分析。 | 是 | 已采用，自动观察优先低成本分析。 |
| 后端按会话节流云端调用。 | 是 | 已采用，画面观察和动作分析都有频率控制。 |
| 限制 WebSocket 帧大小。 | 是 | 已采用，超大帧会被跳过或降级。 |
| 页面不可见、后端离线、AI 播报中暂停或降频。 | 是 | 已采用。 |
| 用户提问触发高优先级视觉上下文。 | 是 | 已采用，视觉/动作类问题会优先刷新上下文。 |
| WebSocket 失败时 HTTP fallback。 | 是 | 已采用，保证演示稳定性。 |
| 场景模式只改 prompt，不增加模型调用次数。 | 是 | 已采用。 |
| 自定义模式限制 prompt 长度。 | 是 | 已采用，前端限制字段长度，后端也裁剪角色卡。 |
| 重新说一遍不重新请求模型。 | 是 | 已采用，只复用本地 TTS。 |
| 流式文字显示不增加额外模型调用。 | 是 | 已采用，同一次模型调用中增量返回。 |
| 历史图片不写入数据库。 | 是 | 已采用，图片进 MinIO，数据库只保存对象引用。 |
| 成本指标默认不打扰用户。 | 是 | 已采用，折叠在“成本与调试”区域。 |
| 用户级预算、调用审计和报表。 | 是 | 暂未实现，作为后续方向。 |

## 9. 关键接口

- `POST /api/vision/analyze`：单帧画面分析。
- `POST /api/vision/analyze-sequence`：多帧动作序列分析。
- `POST /api/conversation/respond`：普通 JSON 对话回复 fallback。
- `POST /api/conversation/respond-stream`：流式文字回复。
- `POST /api/session/end`：结束会话并归档历史。
- `GET /api/session/history`：获取历史会话列表。
- `GET /api/session/history/:sessionId`：获取历史会话详情。
- `GET /api/session/history/:sessionId/final-frame-url`：获取最后一帧预览 URL。
- `WS /api/omni/realtime`：视频关键帧、视觉记忆和对话 WebSocket 通道。

共享类型集中在 `packages/shared`，包括视觉摘要、动作时间线、场景模式、自定义角色卡、会话请求、流式事件、历史会话和 WebSocket 事件。

## 10. 当前限制与后续方向

当前限制：

- 当前是关键帧近实时视觉方案，不是原始连续视频流或 WebRTC 真音视频推流。
- TTS 仍是完整回复生成后再播报，不是流式 TTS。
- 自定义模式只保存在浏览器本地，不支持账号同步。
- 历史记录第一版只读，不支持删除、导出、搜索和筛选。

后续方向：

- 接入云端原生实时音视频多模态协议，进一步提升实时对话感。
- 增加流式 TTS 或云端实时语音输出。
- 增加历史记录删除、导出、搜索和场景筛选。
- 增加用户级预算、调用审计和成本报表。
- 增加端到端演示脚本，覆盖摄像头、麦克风、视觉观察、动作提问、历史归档和降级路径。
- 引入更强动作理解策略，例如更长窗口、多尺度关键帧选择或轻量本地姿态估计。
