# AI 视觉对话助手设计文档

## 1. 项目目标

本项目实现一款 Web 端 AI 视觉对话助手。用户授权摄像头和麦克风后，应用可以预览摄像头画面、识别用户语音或文字输入，并让 AI 结合当前画面、最近动作、会话历史和场景角色进行中文回答，最后通过浏览器 TTS 播报。

当前方案不是上传原始连续视频流，而是采用“本地 2 FPS 采样 + 帧差去重 + WebSocket 流式关键帧 + 后端滑动视觉记忆”的近似视频流方案。浏览器持续采样并只推送有变化的压缩关键帧，后端按会话维护最新画面观察和最近动作变化，云端模型只在节流窗口内分析有价值的帧或短序列。

## 3. 场景角色与体验

场景模式会同时影响右侧展示和后端对话 prompt。每个模式对应一个明确 AI 角色，但不改变视觉采样频率或云端调用频率。

| 场景模式 | AI 角色 | 回答倾向 | 右侧展示重点 |
| --- | --- | --- | --- |
| 通用视觉对话 | 通用视觉对话伙伴 | 先直接回答，再补充不确定点。 | 当前观察、最近变化、推荐追问。 |
| 动作理解助手 | 动作观察员 | 优先用“先……然后……”描述动作变化。 | 最近动作、动作步骤、动作类推荐问题。 |
| 桌面学习助手 | 桌面学习助教 | 先说看到的内容，再给学习或理解建议。 | 桌面内容、学习资料、学习辅助问题。 |
| 演讲/面试练习助手 | 演讲和面试练习教练 | 给一个表现亮点和一个可改进建议。 | 表现反馈、姿态/手势观察、改进建议。 |
| 生活提醒助手 | 生活提醒助手 | 先说观察，再给轻量提醒，不夸大风险。 | 物品变化、注意事项、轻量提醒。 |

角色 prompt 明确禁止在用户可见回复中暴露“视觉摘要”“动作时间线”“基于某个时间”“上下文显示”等工程词，让回答更像自然实时对话。

## 4. 语音体验设计

语音交互沿用浏览器 SpeechRecognition 和浏览器 TTS。本轮增强重点是让用户更清楚地感知 AI 正在做什么：

1. “正在听”：用户可以直接说话，也可以在左侧输入问题。
2. “已听到”：识别到最终语音文本后，短暂显示“已听到：xxx”。
3. “正在思考”：AI 正在结合画面和问题生成回复。
4. “正在回复”：AI 回复文字会边生成边显示，完整生成后再播报，用户可以停止回复或重播上一条回复。

“重新说一遍”只复用上一条 AI 回复文本进行本地 TTS 播报，不新增聊天气泡，也不重新请求模型，因此不会增加云端成本。“继续追问”只聚焦输入框，不自动发送问题，避免误触发额外调用。

## 5. 当前架构

- `apps/web`：React + Vite 前端，负责摄像头预览、本地抽帧、帧指纹去重、WebSocket 关键帧推送、语音识别、TTS、文字提问、场景模式和 UI 展示。
- `apps/server`：NestJS 后端，负责 DashScope 多模态调用、会话回复、关键帧对象存储、session 历史记忆、场景角色 prompt 和 `/api/omni/realtime` WebSocket 视觉记忆代理。
- `packages/shared`：前后端共享类型，包含视觉观察、动作变化、视频流帧、场景模式、Omni WebSocket 事件和对话请求契约。

核心流程：

1. 用户连接设备，前端申请摄像头和麦克风权限。
2. 前端启动预览、浏览器语音识别、文字输入和自动观察。
3. 自动观察在本地采样关键帧，静止或低变化帧只计数不上传。
4. 有变化的关键帧压缩后通过 WebSocket `video_frame` 发送给后端。
5. 后端按 session 维护滑动视觉记忆，并节流生成最新画面观察和最近动作变化。
6. 用户语音或文字提问时，前端携带当前场景；WebSocket 可用时通过 `text` 事件提问，否则回退 HTTP。
7. 后端结合场景角色、视觉记忆、动作变化和历史对话生成自然回复，并通过 HTTP streaming 返回增量文本。
8. 前端流式展示回复，完整生成后用 TTS 播报，用户可停止、重播或继续追问。

## 6. 成本控制策略

| 成本控制技巧 | 是否想到 | 实际采用 |
| --- | --- | --- |
| 不上传原始连续视频，只上传关键帧。 | 是 | 已采用。 |
| 浏览器本地 2 FPS 采样。 | 是 | 已采用，本地采样不等于云端 2 FPS 调用。 |
| 本地帧差去重。 | 是 | 已采用，低变化帧不会进入云端分析。 |
| 压缩关键帧。 | 是 | 已采用，小尺寸 JPEG 和低细节视觉分析。 |
| 默认 `detail: low`。 | 是 | 已采用，自动和手动分析默认低细节。 |
| 后端按会话节流云端调用。 | 是 | 已采用，画面观察和动作变化分别设置分析间隔。 |
| 后端限制最大帧大小。 | 是 | 已采用，超大 WebSocket 关键帧会跳过。 |
| 页面不可见、AI 播报、后端离线时暂停采样或上传。 | 是 | 已采用。 |
| WebSocket 失败时 HTTP fallback。 | 是 | 已采用，保证演示稳定。 |
| 场景模式不增加云端调用。 | 是 | 已采用，只改变 prompt 和 UI 展示重点。 |
| 场景角色 prompt 不增加额外成本。 | 是 | 已采用，只改变同一次对话请求中的系统指令，不增加模型调用次数。 |
| 重新说一遍不重新请求模型。 | 是 | 已采用，只重播本地 TTS。 |
| 流式文本显示不增加额外模型调用。 | 是 | 已采用，同一次对话调用中按增量返回文本。 |
| 成本指标默认不打扰用户。 | 是 | 已采用，右侧“成本与调试”默认折叠。 |
| 用户级预算、调用审计和数据库报表。 | 是 | 暂未实现，作为后续增强。 |

## 7. 关键接口

- `POST /api/vision/analyze`：单帧画面分析，用于手动分析和视觉问题兜底。
- `POST /api/vision/analyze-sequence`：HTTP 多帧动作分析，作为 WebSocket 不可用时的兜底。
- `POST /api/conversation/respond`：HTTP 对话回复兜底，携带 `sceneMode`、视觉观察、动作变化和历史上下文。
- `POST /api/conversation/respond-stream`：HTTP 流式对话回复，使用 `text/event-stream` 返回 `meta`、`delta`、`done` 和 `error` 事件。
- `WS /api/omni/realtime`：流式视觉记忆与对话通道。
  - 客户端事件：`start`、`video_frame`、`vision_context`、`text`、`stop`。
  - 服务端事件：`ready`、`video_summary`、`action_timeline`、`video_status`、`vision_memory_status`、`text`、`error`、`closed`。

## 8. 后续方向

1. 接入 DashScope 原生实时音视频能力，替换当前浏览器语音识别和关键帧近似方案。
2. 增加流式 TTS 或云端实时语音输出，让播报也能边生成边播放。
3. 增加用户级限流、预算告警、调用审计和成本报表。
4. 将会话、消息、视觉观察、动作变化和指标持久化到数据库。
5. 增加端到端演示脚本，覆盖摄像头授权、WebSocket 观察、语音提问、动作问题和降级路径。
6. 引入更强的动作理解策略，例如更长窗口、多尺度关键帧选择或轻量本地姿态估计。
