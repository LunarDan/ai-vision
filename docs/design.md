# AI 视觉对话助手设计文档

## 1. 项目目标

本项目实现一款 Web 端 AI 视觉对话助手。用户授权摄像头和麦克风后，应用可以预览摄像头画面、识别用户语音，并让 AI 结合当前画面与最近几秒动作进行中文回答，最后通过浏览器 TTS 播报。

当前方案不是上传原始连续视频流，而是采用“本地 2 FPS 采样 + 帧差去重 + WebSocket 流式关键帧 + 后端滑动视觉记忆”的近似视频流方案。浏览器持续采样并只推送有变化的小图关键帧，后端按会话维护最新画面摘要和最近动作时间线，云端模型只在节流窗口内分析有价值的帧或短序列。

## 3. 当前架构

- `apps/web`：React + Vite 前端，负责摄像头预览、本地抽帧、帧指纹去重、WebSocket 关键帧推送、语音识别、TTS 和 UI 展示。
- `apps/server`：NestJS 后端，负责 DashScope 多模态调用、会话回复、MinIO 关键帧存储，以及 `/api/omni/realtime` WebSocket 视觉记忆代理。
- `packages/shared`：前后端共享类型，包括视觉摘要、动作时间线、视频流帧、Omni WebSocket 事件和对话请求契约。

核心流程：

1. 用户点击连接，前端申请摄像头和麦克风权限。
2. 前端启动预览、浏览器语音识别和自动观察。
3. 自动观察每 500ms 在本地采样一帧，并计算轻量帧指纹。
4. 静止或低变化帧只在本地计数，不上传。
5. 有变化的关键帧被压缩成小尺寸 JPEG，通过 WebSocket `video_frame` 事件发送给后端。
6. 后端按 session 维护最近约 6 秒关键帧滑动窗口，并节流调用多模态模型生成：
   - `video_summary`：最新画面摘要。
   - `action_timeline`：最近动作时间线。
   - `video_status`：流式帧数、缓冲帧数、云端分析次数。
7. 用户语音提问时，如果 WebSocket 可用，前端优先通过 `/api/omni/realtime` 发送 `text` 事件，让后端直接读取当前 session 的视觉记忆回答。
8. 如果 WebSocket 不可用或超时，前端回退到 HTTP `/api/conversation/respond`，并携带本地缓存的视觉摘要和动作时间线。

## 4. 成本控制策略

| 成本控制技巧 | 是否想到 | 实际采用 |
| --- | --- | --- |
| 不上传原始连续视频，只上传关键帧 | 是 | 已采用。当前没有向云端传原始视频流。 |
| 浏览器本地 2 FPS 采样 | 是 | 已采用。2 FPS 只发生在本地，云端不会按 2 FPS 调用。 |
| 本地帧差去重 | 是 | 已采用。低变化帧不进入动作缓冲，也不推送 WebSocket。 |
| 压缩关键帧 | 是 | 已采用。动作流关键帧压缩为小尺寸 JPEG；单帧分析压缩到 768px 宽、JPEG 0.72。 |
| 默认低细节视觉分析 | 是 | 已采用。自动与手动画面分析默认 `detail: low`。 |
| 后端按会话节流云端调用 | 是 | 已采用。画面摘要和动作时间线分别设置分析间隔，避免每帧调用模型。 |
| 后端限制最大帧大小 | 是 | 已采用。WebSocket 关键帧超过大小限制会被跳过。 |
| 页面不可见、AI 播报、后端离线时暂停采样/上传 | 是 | 已采用。前端根据状态暂停自动观察和动作采样。 |
| WebSocket 失败时 HTTP fallback | 是 | 已采用。保证演示稳定，不因流式通道失败中断连续对话。 |
| 用户级预算、服务端持久化审计 | 是 | 暂未实现。后续可加入用户维度配额、成本面板和数据库审计。 |
| 云端原生实时音视频接口 | 是 | 暂未采用。需要确认 DashScope Omni 实时音视频协议、权限和成本后再接入。 |

## 5. 关键接口

- `POST /api/vision/analyze`：单帧画面分析，作为手动分析和视觉问题兜底。
- `POST /api/vision/analyze-sequence`：HTTP 多帧动作分析，作为 WebSocket 不可用时的兜底。
- `POST /api/conversation/respond`：HTTP 对话回复兜底。
- `WS /api/omni/realtime`：流式视觉记忆与对话通道。
  - 客户端事件：`start`、`video_frame`、`vision_context`、`text`、`stop`。
  - 服务端事件：`ready`、`video_summary`、`action_timeline`、`video_status`、`text`、`error`、`closed`。

## 6. 后续方向

1. 接入 DashScope 原生实时音视频能力，替换当前浏览器语音识别和关键帧近似方案。
2. 增加服务端用户级限流、预算告警、调用审计和成本报表。
3. 将会话、消息、视觉摘要、动作时间线和指标持久化到数据库。
4. 增加端到端演示脚本，覆盖摄像头授权、WebSocket 视频流、语音提问、动作问题和降级路径。
5. 引入更强的动作理解策略，例如更长窗口、多尺度关键帧选择或轻量本地姿态估计。
